Loading partials & modules

A wrapper with a src="" attribute auto-loads on connect. File extension picks the loader: .html swaps the response into the wrapper and wakes its bindings; .js / .mjs are loaded with native import().

The nav sidebar at the top of this page and this very section are both loaded from /partials/. Open Network → Fetch/XHR in DevTools and refresh.

HTML partial

How the nav is loaded 
<data-wrapper id="nav" src="/partials/nav.html" data-nav-open="false"
    onload="this.register({
      'nav/toggle': () => this.put('navOpen', v => !v),
      'nav/close':  () => this.put('navOpen', false),
      /* … */
    })"></data-wrapper>

State (data-nav-open) and handlers (registered via onload) live with the wrapper. The partial contributes only the view — no <data-wrapper> tag inside, just the children. onload fires on connect, so handlers register before the partial's @ bindings wire up.

JS module

Module shape — src="/controllers/widget.js" 
export default function init(wrapper) {
  wrapper.register({
    'thing/do': () => wrapper.put('count', n => Number(n) + 1),
  });
}

Loaded with native dynamic import() — never eval, never new Function(). Async initializers are awaited.

Notes

  • data:load event fires on the wrapper after the resource is applied.
  • HTML inserted via innerHTML — <script> tags do not execute (HTML5 spec).
  • Cross-origin URLs are blocked by the browser's same-origin / CORS rules — no app-level policy needed.
  • This feature is for trusted, app-authored sources. Don't src="" arbitrary user URLs.
