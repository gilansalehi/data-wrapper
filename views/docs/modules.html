Cross-module imports

data-module marks the component module in a view and gives it a stable import name. The loader registers the inline script's content under that name in an import map; other modules then reference it like any ES module.

Producer

<!-- /views/nav.html -->
<script type="module" data-module="@view/nav">
    export let open = false;
    export function toggle() { open = !open; }
</script>

Consumer

<!-- /views/dashboard.html -->
<script type="module" data-module="@view/dashboard">
    import { open } from '@view/nav';
    export const navStateLabel = () => open ? 'visible' : 'hidden';
</script>

<p>Nav is <span $text="navStateLabel"></span></p>

ESM live bindings make this work without the framework doing anything special: open reads through to the producer module's latest value, and when an action there mutates it, every consuming runtime re-derives on the next flush.

This is the preferred cross-wrapper communication channel. Template bindings can also read another mounted wrapper explicitly with //id/name, but that form is an escape hatch for DOM-hosted composition. Shared state belongs in modules.

Plain state modules

Modules that aren't loaded as views can live anywhere on disk. The framework's only requirement is that the importing view can resolve the path. With the import map shown below, a state module at /state/todos.js is imported as state/todos.js: 

<!-- in the host page -->
<script type="importmap">{"imports":{"state/":"/state/"}}</script>
// inside any component module
import { todos, addTodo } from 'state/todos.js';

Naming rules

  • data-module marks the inline component module, and the value must not be empty.
  • The name must be unique per document. Two views with the same name throw a duplicate-registration error.
  • @view/* and @docs/* are conventions; the name itself is any valid ESM specifier.
