Cross-module imports

Module imports are the best way to share application state between views. A view script with data-module gets a stable module name, and other component modules can import from that name just like they import from any browser module.

The first view is the producer. It owns the state and exports the values or functions that other views should be allowed to use. 

<script type="module" data-module="@view/nav">
    export let open = false;
    export function toggle() {
        open = !open;
    }
</script>

The second view is the consumer. It imports from the producer module and can expose imported functions to its own template. In this example, the dashboard reads open and re-exports toggle so its button can call the shared function. 

<script type="module" data-module="@view/dashboard">
    import { open, toggle } from '@view/nav';

    export const navLabel = () =>
        open ? 'Navigation open' : 'Navigation closed';

    export { toggle };
</script>

<button @click="toggle">Toggle nav</button>
<p $text="navLabel"></p>

ES module imports are live bindings. The dashboard does not receive a stale copy of open; each time navLabel runs, it reads the current value from @view/nav. When toggle changes that value from an event binding, data-wrapper flushes the mounted runtimes and the dashboard output updates.

The producer must already be registered before a consumer imports it. The example below shows the intended order, but wrapper loads are still asynchronous. If several consumers need shared state during their first import, a plain state module with a host import map is usually simpler. 

<!-- register @view/nav before dashboard.html imports it -->
<data-wrapper src="nav.html"></data-wrapper>
<data-wrapper src="dashboard.html"></data-wrapper>

Use this pattern for normal shared state. Template bindings can also read another mounted wrapper with //id/name, but that is a DOM escape hatch for narrow cases. If the state belongs to the application instead of one rendered wrapper, put it in a module and import it.

Plain state modules

A shared module does not have to be a view. Plain JavaScript modules can live anywhere the browser can resolve them. If you want a short import prefix for those files, add a normal import map on the host page. 

<script type="importmap">
{
    "imports": {
        "state/": "/state/"
    }
}
</script>

The framework does not need to know about plain state modules. They are just browser modules. The only requirement is that the importing view can resolve the specifier. 

export let todos = [];

export function addTodo(task) {
    todos = [...todos, { id: Date.now(), task, done: false }];
}
import { todos, addTodo } from 'state/todos.js';

Naming rules

The data-module value is both the marker that tells data-wrapper which script to load and the name other modules import from. Keep names stable and unique.

  • data-module marks the inline component module, and the value must not be empty.
  • A loaded document can only register one view for a given name. Reusing the same name for a different view throws a duplicate-registration error.
  • @view/* and @docs/* are conventions. The name itself can be any valid module specifier your app chooses.
