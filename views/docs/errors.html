Error handling

The framework throws on failure rather than swallowing errors. A failed fetch, an unparseable view, a blank or duplicate data-module, a factory that throws, an unknown *directive, or an action that throws all propagate as an exception — an uncaught promise rejection on the async load path. Framework-raised loader errors name the view URL, and the custom element wraps a failed load with its src; an unknown *directive names the directive. An error thrown by your own factory or action code propagates as-is — surfaced without rewriting.

The one deliberate non-throwing case is a lookup miss: a $ or * binding whose name resolves nowhere renders the literal name and emits a console.warn. That is almost always a typo, but it is non-fatal and never aborts the bindings around it.

There is no separate error event or error-state model — the mechanism is a thrown error, observable in the console and through global handlers such as window.onunhandledrejection.
