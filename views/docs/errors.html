Error handling

The framework throws on failure rather than swallowing errors. A failed fetch, an unparseable view, a missing or duplicate data-module, a factory that throws, an unknown *directive, or an action that throws all propagate as an exception — an uncaught promise rejection on the async load path. Each error carries the view URL, the wrapper's src, or the binding/action name, so the source is identifiable.

The one deliberate non-throwing case is a lookup miss: a $ or * binding whose name resolves nowhere renders the literal name and emits a console.warn. That is almost always a typo, but it is non-fatal and never aborts the bindings around it.

There is no separate error event or error-state model — the mechanism is a thrown error, observable in the console and through global handlers such as window.onunhandledrejection.
