Install & scripts

The fastest way to try data-wrapper is to load the browser bundle from a CDN. Once the script is on the page, every <data-wrapper> element can load a view file with its own HTML and component module. 

<script src="https://unpkg.com/data-wrapper/dist/data-wrapper.min.js"></script>

If your component modules only use local exports, that script tag is enough. If a component imports public helpers such as action, flush, DW_FORMATTERS, or types from data-wrapper, add an import map before the framework script so the browser knows where that module name points. 

<script type="importmap">
{
    "imports": {
        "data-wrapper": "/dist/data-wrapper.js"
    }
}
</script>
<script src="/dist/data-wrapper.js" type="module"></script>

Package installs use the same module name. Importing data-wrapper registers the custom element. Named imports give component modules access to the public reactivity and extension APIs. 

npm install data-wrapper
# or
bun add data-wrapper
import 'data-wrapper';

import { action, flush, DW_FORMATTERS } from 'data-wrapper';

Browser support

data-wrapper targets modern evergreen browsers with native ES modules, custom elements, and import maps. The framework registers component modules at runtime, so component code can use normal import syntax.

Some browsers need help with runtime import maps. For those browsers, add a data-shim-src attribute that points to es-module-shims. The shim loads only when native module resolution fails, so browsers with native support do not download it. 


<script src="/dist/data-wrapper.js" type="module"
    data-shim-src="https://ga.jspm.io/npm:es-module-shims@2/dist/es-module-shims.js">
</script>

A CDN URL is the simplest shim source. Vendoring a copy works the same way. If a module cannot be resolved and no shim is configured, the loader throws a clear error that points back to this browser support setup.
