Reactivity model

The framework has exactly two primitives — action(), a function wrapper, and flush(), a manual flush trigger. Every @event binding is auto-wrapped, so most components never have to call either directly.

The action boundary

When an action runs, every active ComponentRuntime re-reads its subscribed outputs and publishes the ones that changed (compared with Object.is). Calls coalesce inside a microtask, so a chain of nested action invocations triggers one flush at the end, not one per call.

action()

Wrap functions that mutate module state but get called from outside an @event boundary — imported writers, timer callbacks, WebSocket handlers, anything async that runs after the next microtask. 

import { action } from 'data-wrapper';

// single function
export const setFilter = action(f => { filter = f; });

// object — wraps each value, returns the same shape
export const { addTodo, removeTodo } = action({
    addTodo:    item => { todos = [...todos, item]; },
    removeTodo: id   => { todos = todos.filter(t => t.id !== id); },
});

action() is idempotent — wrapping an already-wrapped function returns it unchanged, so passing the same writer through multiple layers doesn't multiply flushes.

Async actions

If the wrapped function returns a Promise, the action flushes again when the promise resolves. The synchronous portion's mutations publish before the first await; post-await mutations publish when the chain settles. 

export const loadItems = action(async () => {
    loading = true;
    items   = await fetch('/items').then(r => r.json());
    loading = false;
});

flush() as an escape hatch

For mutations that can't be wrapped at the call site — a setInterval, a third-party event listener — call flush() manually after the mutation. 

import { flush } from 'data-wrapper';

setInterval(() => {
    clock = new Date();
    flush();
}, 1000);
