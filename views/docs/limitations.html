Known limitations

The lib is pre-1.0. The items below are tracked, intentional gaps — either deferred features or accepted limitations with a documented scope. Each will either ship resolved in 1.0 or carry a final documented status.

*if clone cleanup

When a *if body is removed, subscriptions created inside that body stay registered on the nearest row or wrapper until that owner is itself destroyed. Repeatedly toggling a long-lived *if accumulates stale subscriptions. They are functionally inert — they fire harmless updates against detached DOM — but the memory growth is real. Tracked as a 1.0 fix.

Cross-wrapper host limits

//id/path is supported as an explicit read from another loaded wrapper's component scope. It is intentionally narrow: it does not dispatch actions, subscribe to wrapper load/reload, inspect row scopes, or retry if the target appears later. Use module imports for normal shared state.

Reserved binding syntax

Protocol-prefixed values are still reserved for future custom resolvers. Using them today is a no-op: they do not render static literals, dispatch parsed-path events, run directives, or create child input values.

  • Protocol-prefixed values (localStorage://, etc.) — reserved for custom resolvers.

No lifecycle hook matrix

The default factory is the mount hook — it runs once per wrapper. Use context.cleanup() for per-wrapper teardown work such as timers, observers, or external listeners. The framework does not currently expose a broader lifecycle matrix such as before-update, after-render, or before-unmount hooks.

No SSR or hydration

The framework is client-rendered. There's no server-side render path, no hydration protocol, and no plan to add either before 1.0. First contentful paint depends on the bundle loading and each wrapper's first load() call.
