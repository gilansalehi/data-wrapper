Three tokens

A view file is regular HTML, so data-wrapper uses attributes to connect that HTML to your module exports. Each special attribute starts with one of three tokens. The token tells data-wrapper what kind of work the attribute should do.

Token Purpose How to...
$ Data Binding $prop="name" reads a binding and writes it to a DOM property or attribute.
* Directives *directive="name" passes a binding to a structural directive such as *list or *if.
@ Event Handling @event="action" listens for a browser event and calls an exported action.

The value on the right side of the attribute is a binding name. In the simplest case, that name points to a value or function exported by the component module. This is the same counter from the intro, now shown as a token example. 

<script type="module" data-module="@view/counter">
    export let count = 0;
    export const doubled = () => count * 2;
    export function inc(event) { count += Number(event.target.value); }
</script>

<button @click="inc?prevent" value="1">+1</button>
<output $text="count"></output>
<output $text="doubled"></output>

The $text attributes read exported values and write them to DOM text. The @click attribute listens for a click and calls the exported inc function. When inc changes count, data-wrapper re-reads the connected bindings and updates the outputs.

Directives use the same binding names, but they control structure instead of writing one property. A *list directive reads an array and renders its template once for each item. Inside that template, paths that start with ./ read from the current row item. 

<script type="module" data-module="@view/todos">
    export let todos = [
        { id: 1, task: 'Try $ bindings', done: true },
        { id: 2, task: 'Render a *list', done: false },
        { id: 3, task: 'Call an @ action', done: false },
    ];
    export const emptyTodos = () => todos.length === 0;

    export function toggle(event) {
        const { item } = event.detail;
        todos = todos.map(todo =>
            todo.id === item.id ? { ...todo, done: !todo.done } : todo
        );
    }
</script>

<template *list="todos">
    <label>
        <input type="checkbox" $checked="./done" @click="toggle">
        <span $text="./task"></span>
    </label>
</template>

<template *if="emptyTodos">
    <div>All Done!</div>
</template>

In this example, *list="todos" reads the exported array. The checkbox and span are rendered inside a row, so ./done and ./task read fields from the current todo item. The toggle action still comes from the module, and the event detail includes the row item that was clicked. The *if directive reads emptyTodos and renders the placeholder only when the list is empty.

Binding names resolve from the place where the DOM node is rendered. By default, a binding will start in the local scope and climb up until it finds a match. Use standard tree traversal syntax to specify a binding that should read from the nearest row, a parent row, the component root, or another wrapper.

  • view checks the nearest row, outer rows, the wrapper instance, and then named module exports.
  • ./done reads only from the nearest surrounding *list item.
  • ../order reads only from the parent row. Repeat ../ to move higher.
  • /view reads from the wrapper's component root and bypasses rows.
  • //cart/total reads from another loaded wrapper by DOM id.
