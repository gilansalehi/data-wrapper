Binding contexts

A binding resolves from the place where its DOM node is rendered. The wrapper starts with a root context, which is backed by the component module and any object returned by the default factory. Each *list then creates a row context for the item it renders. Nested lists create nested row contexts.

This is what makes the same binding syntax work at different depths in the DOM. In the example below, the outer template renders orders and the inner template renders line items inside each order. 

<script type="module" data-module="@view/orders">
    export const storeName = 'Downtown';
    export const orders = [
        {
            id: 1,
            customer: 'Ada',
            lines: [
                { id: 'a', sku: 'Notebook', qty: 2 },
                { id: 'b', sku: 'Pencil', qty: 6 },
            ],
        },
    ];
</script>

<template *list="orders">
    <article>
        <h3 $text="./customer"></h3>
        <p>Store: <span $text="storeName"></span></p>

        <template *list="./lines">
            <p>
                <span $text="../customer"></span>
                ordered
                <span $text="./qty"></span>
                <span $text="./sku"></span>
            </p>
        </template>
    </article>
</template>

The first *list reads orders from the wrapper root and creates one row context per order. Inside that row, ./customer reads the current order. The bare name storeName starts at the current row, does not find a field with that name, and then climbs to the wrapper root.

The inner *list="./lines" starts from the current order and renders one row for each line item. Inside that inner row, ./qty and ./sku read the line item, while ../customer skips the line item and reads the parent order.

Resolution rules

These forms are available in $ bindings, * directive sources, and child-wrapper input expressions. They all follow the same lookup rules.

FormResolves to
nameStarts at the nearest row that owns the name, then climbs through outer rows, then reads the wrapper root. At the root, factory instance bindings win over named module exports.
./nameReads the nearest enclosing *list row only.
../nameReads the parent row only. Repeat the prefix, as in ../../name, to target higher rows.
/nameReads the wrapper root and bypasses every row context.
//id/nameReads another loaded <data-wrapper id="id">, root scope only.

Parent-row paths are exact. If ../name targets a parent row that does not own name, it does not keep climbing and it does not fall through to the wrapper root. That keeps nested templates predictable.

Cross-wrapper reads are explicit and root-only. They use document.getElementById(), require a loaded <data-wrapper>, and do not inspect another wrapper's row scopes. Use module imports for normal shared state; use //id/name only when the DOM relationship is the point.
