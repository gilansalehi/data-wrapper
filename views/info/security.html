Security

data-wrapper's security posture is measured against published standards rather than an ad-hoc list: the OWASP DOM-Based XSS Prevention Cheat Sheet, the CWE-79 weakness taxonomy, CSP Level 3, and the Trusted Types API. This section states what the framework guarantees, what it leaves to the application, and where it does not yet reach.

Scope & threat model

The current contract assumes developer-authored views and trusted bound data. Rendering user-generated content (UGC) and loading third-party views are out of scope today: an app that renders untrusted content owns its sanitization until the Trusted-Types tier lands (see below). Under that contract the only untrusted bytes are whatever the app's own backend returns, and encoding those is an application responsibility — the same default posture as mainstream client frameworks.

Default protections

  • Safe-by-default text. $text writes textContent; bound values are never parsed as HTML.
  • Gated raw HTML. $innerhtml and $outerhtml throw. Raw HTML requires the explicit $unsafeHTML opt-in, so the dangerous sink can never be reached by accident.
  • URL-scheme neutralization. javascript: and vbscript: values are dropped from href, src, action, and formaction.
  • No string evaluation. No eval, Function, or string-argument timers. Events bind functions, not code strings; JSON uses JSON, not eval.
  • Same-origin view loading by default. A <data-wrapper src> loads same-origin unless an explicit policy allows other origins, limiting injected-src escalation.

Application responsibilities

Within the stated scope the framework closes the common DOM-XSS gotchas; these remain the application's to own:

  • Do not pass untrusted content to $unsafeHTML or to an <iframe $srcdoc> — both are raw-HTML sinks by definition.
  • Sanitize or encode backend data before binding it into a URL or HTML sink; the "trusted data" assumption must actually hold.
  • Set defense-in-depth the framework cannot: HttpOnly cookies, Subresource Integrity on the CDN framework script, and the CSP directives noted above.
