Why Choose Data-Wrapper

Most frameworks ask you to move your UI into JavaScript. Components replace HTML. Templates become a dialect of JSX. State lives in a parallel model, mirrored into the DOM by a render function that runs on every change.

That approach works — and it costs you the platform. Build pipelines. Hydration boundaries. A second mental model for what is on screen. Tooling for the tooling.

data-wrapper goes the other direction. State lives where you can already read it: in data-* attributes. One custom element observes its own attributes, compiles bindings once at wake time, and broadcasts updates to pre-arranged subscriber closures. No virtual DOM. No compiler. No build step.

What you keep

  • HTML as the source. The markup you wrote is the page that ships.
  • The browser as the runtime. URL parsing, attribute observation, native event bubbling — all already there, already fast.
  • State you can inspect. Attributes are visible in DevTools, mutable from the inspector, and reactive to CSS selectors for free.
  • Server HTML that starts alive. No hydration step; the wrapper reads its own attributes on connect.

What you skip

  • JSX, build tools, and compilers
  • Virtual DOM diffing and reconciliation trees
  • A parallel state model that has to stay in sync with the DOM
  • Component libraries you have to learn before you can ship anything

Where it fits

Internal tools. Dashboards. Embedded widgets that have to coexist with someone else's framework. Progressive enhancement of server-rendered pages. Anywhere you want a predictable reactive layer on top of HTML the server already produces.

Sub-kilobyte runtime. Three tokens to learn — $ for data, * for structure, @ for events. State you can read with your eyes.

data-wrapper does not try to replace React for app-shell scenarios. It asks whether you needed React at all.
