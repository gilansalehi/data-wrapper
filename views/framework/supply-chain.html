Zero dependencies, zero supply chain

Install a frontend framework and you install its dependency tree: hundreds of packages, written by people you will never meet, running with your build's privileges on every developer machine and every CI runner. You audited the framework. You did not audit the 300 packages underneath it. Nobody does.

data-wrapper has no dependencies. It is one file. The supply chain is two parties: you, and the browser you already trust.

Why a dependency is a liability

  • It is code you run but did not read. Every package executes with your build's privileges, and most of the tree you have never opened.
  • Trust is transitive. You vet a library; you do not vet the packages underneath it, or the packages underneath those.
  • The human layer is the soft target. Accounts get phished, maintainers burn out and hand off keys, domains expire and get re-bought.
  • postinstall runs arbitrary code. Every npm install can execute scripts on every machine that runs it, before you have run a line of your own.
  • The blast radius is the whole ecosystem. One popular package compromises thousands of apps in a single release.

What “no dependencies” actually buys you

One file to vendor

Copy data-wrapper.js into your project and pin it by hash. There is no tree behind it — what you read is what you ship.

Nothing transitive to audit

No lockfile to diff, no advisory feed to chase, no Dependabot noise. The audit surface is a single file you can read in one sitting.

No install step, no install script

There is no postinstall hook because there is no install. You add a <script> tag; the browser does the rest.

Other people's advisories

When the next npm package is compromised, the question “are we affected?” has a fast answer: no. It was never in the build.

Track it yourself

Hacker News is where the post-mortems get argued; for the authoritative record, watch the advisory databases directly: GitHub Advisory Database, OSV, and Socket. Then ask how much of that feed you would like to be exposed to.

This is the same bet the rest of the framework makes: the less you depend on, the less can be taken from you. The constraints are the feature — and an empty dependency list is the strongest constraint of all.
