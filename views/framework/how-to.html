How To

Awaken your HTML with familiar concepts:

  • Wrap it with a <data-wrapper> to awaken the DOM

    One <data-wrapper> web component to bind it all together.

  • pURLs track your data like a URL:

    dwrl://data-wrapper/user/name/first?format=upper
    Protocol

    dwrl:// reads from a wrapper's state. Other protocols like api:// for fetch-driven injectors are on the roadmap.

    Host

    Names the target wrapper. Today this is the closest <data-wrapper> ancestor; cross-wrapper mesh resolution by id is on the roadmap.

    Path

    Slash-separated drill into state. The first segment hits the state proxy (JSON parse at the dataset boundary); subsequent segments walk the parsed tree.

    Query

    Framework flags. ?format= pipes the value through formatters; ?key= sets *list row identity; ?prevent / ?stop / ?immediate control @-event dispatch.

    Hover or focus a colored segment — the familiar URL, supercharged for page-level reactivity.

  • Custom attribute tokens awaken the DOM:

    • $ sets a value
    • @ emits an event
    • * controls the DOM

How to think in data-wrapper

If you know React, the mental model is familiar: state changes, the UI updates. The difference is where the model lives. You do not move your HTML into JavaScript. You put a reactive boundary around real HTML and let the browser do the work it already knows how to do.

1. State is visible

A wrapper's data-* attributes are its state. That means initial state can ship in HTML, CSS can react to it, and DevTools can inspect it without framework plugins. 

<data-wrapper id="counter" data-count="0">
  <output $text="/count"></output>
</data-wrapper>

2. HTML declares subscriptions

The $ token binds wrapper state to DOM properties. Use the property name you already know: $text, $value, $checked, $class, $open. 

<button $disabled="/saving?format=bool">Save</button>
<p $text="/message"></p>

3. Events stay browser-native

The @ token listens for native events and emits a named action topic. Your JavaScript handles behavior; your HTML keeps the wiring readable. 

<button @click="count/inc">+</button>

<script>
counter.register({
  'count/inc': () => counter.put('count', n => Number(n) + 1),
});
</script>

Structure is just another subscription

The * token is for DOM structure. *if mounts a node when a value is truthy. *list renders rows from a template and keeps row updates scoped to the row data. 

<p *if="/ready">Ready.</p>

<ul *list="/todos">
  <template>
    <li>
      <input type="checkbox" $checked="./done" @change="todo/toggle">
      <span $text="./task"></span>
    </li>
  </template>
</ul>

DWRL: paths that read like URLs

DWRL is the lookup syntax behind the tokens. It deliberately looks like a URL because the browser already knows how to parse URLs. The token says what to do; the DWRL says what data to use.

DWRLMeaningExample
/count Read wrapper state. $text="/count"
./task Read the current row inside a *list. $text="./task"
?format=name Pipe the value through a formatter. $text="/total?format=currency"
?key=uuid Choose the row identity key for list reconciliation. *list="/users?key=uuid"

The whole loop

A wrapper wakes its subtree once, compiles @, $, and * attributes into subscriptions, then future updates are cheap subscriber calls. No virtual DOM. No hooks. No dependency arrays. No build step. The DOM remains the source, and the wrapper is the small reactive engine around it.
