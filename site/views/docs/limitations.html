Known limitations

data-wrapper is still pre-1.0, so a few boundaries are intentional. These are not hidden features or special cases to depend on. They are the edges of the current public contract.

Cross-wrapper reads are narrow

//id/path is an explicit read from another loaded wrapper's root component scope. It does not inspect row scopes, dispatch actions, subscribe to wrapper load or reload, or retry if the target appears later. Use module imports for normal shared state.

View-module imports depend on registration order

A view can import another view by its data-module name only after that producer has been loaded and registered. If several components need shared state during their first import, put that state in a plain JavaScript module and expose it with a host import map.

Protocol resolvers are reserved

Protocol-prefixed values such as localStorage://key are reserved for future source resolvers. Today they are inert: they do not render static literals, dispatch parsed-path events, run directives, or create child input values.

Custom directives cannot create row scopes

Custom directives can decorate DOM, respond to a bound value, create DOM under the context they receive, and register cleanup. They do not currently get a public helper for creating new row contexts. Scope- introducing behavior remains built into *list.

There is no lifecycle hook matrix

The default factory is the mount hook — it runs once per wrapper. Use context.cleanup() for per-wrapper teardown work such as timers, observers, or external listeners. The framework does not currently expose a broader lifecycle matrix with hooks such as before-update, after-render, or before-unmount hooks.

Keyless lists match rows by value

*list identifies rows by a key field — id by default, or a chosen field with *list="items?key=field". Without a key field, a row's identity falls back to a structural JSON snapshot of the item. Two equal items then collapse to a single row, and editing any field changes that row's identity, forcing a teardown and rebuild instead of an in-place update. Supply a stable key whenever list items can repeat or change over time.

Bindings do not run inside SVG subtrees

The current wake pass skips SVG descendants. This avoids treating SVG DOM properties like regular HTML properties; for example, SVG className is not the same writable string property as an HTML element's className. Inline SVG is fine as static markup, but SVG-specific binding support is a future feature.

There is no SSR or hydration path

data-wrapper is client-rendered. There is no server-side render path and no hydration protocol. First contentful paint depends on loading the bundle and each wrapper's first load() call.
