Module scope vs. default factory

A component view can expose bindings from two places. Named exports live in the module, and the module is shared by every wrapper that loads it. A default factory can create a separate instance object for one mounted wrapper. Most views can use named exports only; reach for a factory when each wrapper needs its own copy of some state.

Named exports are a good fit for shared application state. If two wrappers load this view, both wrappers read and update the same theme export. 

<script type="module" data-module="@view/theme">
    export let theme = 'light';
    export const label = () => `Theme: ${theme}`;

    export function toggleTheme() {
        theme = theme === 'light' ? 'dark' : 'light';
    }
</script>

<button @click="toggleTheme">Toggle</button>
<output $text="label"></output>

That shared behavior is often exactly what you want. A theme switcher, a user session, or a shared collection should usually have one source of truth. But a counter, form, or reusable card often needs state that belongs to one wrapper only.

A default factory handles that second case. The factory runs once for each mounted wrapper. It receives a context object, and it may return an object of instance bindings for that wrapper. The instance object sits in front of the module exports, so an instance binding with the same name wins. 

<script type="module" data-module="@view/counter">
    export const label = 'Counter';

    export default ({ props }) => {
        let count = Number(props.start ?? 0);

        return {
            get count() { return count; },
            increment() { count += 1; },
        };
    };
</script>

<h3 $text="label"></h3>
<button @click="increment">+1</button>
<output $text="count"></output>
<data-wrapper src="counter.html?start=5"></data-wrapper>
<data-wrapper src="counter.html?start=20"></data-wrapper>

Both wrappers use the same module export for label, but each wrapper gets its own count variable and its own increment function. The query string supplies the starting value for that specific wrapper.

Child inputs

The factory context includes props. Props come from the query string on the wrapper's src. When a child wrapper is mounted inside another view, those query entries are resolved from the place where the child appears in the parent DOM. 

<script type="module" data-module="@view/orders">
    export const customer = { firstName: 'Ada', lastName: 'Lovelace' };
    export let orderStatus = 'Ready';
</script>

<data-wrapper src="card.html?customer&status=orderStatus"></data-wrapper>

The shorthand ?customer means ?customer=customer. It asks the child to receive a prop named customer, resolved from the parent binding with the same name. The status=orderStatus entry gives the child a prop named status, but reads it from the parent's orderStatus binding. 

<script type="module" data-module="@view/card">
    export default ({ props }) => ({
        customer: props.customer,
        status: props.status,
    });
</script>

<h3 $text="customer/firstName"></h3>
<p $text="status"></p>

A resolved prop is a stable function reader. In the card example, the factory returns props.customer and props.status directly, so the card template keeps reading the current parent values. If the factory called props.customer() instead, it would snapshot the current value at mount time.

A query value that does not resolve to a binding becomes a static input. Static inputs are parsed with JSON.parse when possible, so start=5 becomes the number 5, enabled=true becomes true, and encoded JSON can pass arrays or objects. Values that are not valid JSON, such as theme=light or file=/views/demo.html, stay as strings. If a value should be the string "true" instead of the boolean true, quote it as JSON in the URL.

Inputs are not automatically added to the child template. The factory chooses which props become bindings by returning them. This keeps child inputs explicit: the query string passes values into the module, and the factory decides what the HTML can bind to.

Child inputs use the same lookup forms as template bindings. Use ./name for the nearest row, ../name for a parent row, /name for the wrapper root, and //id/name for an explicit read from another loaded wrapper. Protocol-prefixed inputs are reserved for future resolvers and are omitted today.

The ComponentContext shape

The factory receives one frozen object. The most common field is props, but the full context is available when a component needs the wrapper element, the resolved URL, or teardown.

FieldTypePurpose
wrapperHTMLElementThe mounted <data-wrapper> element itself.
urlURLThe full resolved src URL for this wrapper.
paramsURLSearchParamsThe raw query parameters from the resolved src.
propsReadonly objectResolved input values or function readers. It also includes reserved props.url, the original src string.
cleanupfunctionRegisters teardown work owned by this wrapper instance.
