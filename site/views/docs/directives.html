Directives

A directive reads a binding and lets JavaScript decide what should happen to the DOM. The built-in directives are *list, *if, and *src. They are written on <template> elements because the template body is the content that data-wrapper inserts, updates, projects, or removes. 

<script type="module" data-module="@view/tasks">
    export const tasks = [
        { id: 1, label: 'Read the docs', done: true },
        { id: 2, label: 'Try directives', done: false },
    ];
</script>

<ul>
    <template *list="tasks?key=id">
        <li $class="./done?bool=done:active">
            <span $text="./label"></span>
            <template *if="./done">
                <strong>done</strong>
            </template>
        </li>
    </template>
</ul>

*list reads an array and renders the template body once for each item. Each rendered item gets its own row context, so ./label and ./done read from the current task. By default, list reconciliation uses each item's id; pass ?key=field when a different field should identify the row.

*if shows its template body when the bound value is truthy and removes it when the value is falsy. It does not create a new row context. In the example above, *if="./done" still reads from the surrounding *list item.

Custom directives

Custom directives are registered in DW_DIRECTIVES. The name after the * token selects the directive, and the value on the right side is the binding the directive will receive. A custom directive returns an updater function; data-wrapper calls that updater immediately and again whenever the bound value changes. 

import { DW_DIRECTIVES } from 'data-wrapper';

DW_DIRECTIVES.set('badge', ({ el, cleanup }) => {
    const badge = document.createElement('small');
    badge.className = 'badge';
    el.append(badge);

    cleanup(() => badge.remove());

    return value => {
        badge.textContent = value == null ? '' : String(value);
    };
});
<button *badge="unreadCount">Inbox</button>

The framework owns the source subscription. The directive owns anything it creates: extra DOM, timers, observers, or event listeners. Register that teardown work with cleanup() so it runs when the surrounding wrapper or row is removed.

If a directive creates DOM that contains data-wrapper bindings, call wake(node, ctx) with the context the directive received. That wires the new DOM under the same scope as the directive host. 

DW_DIRECTIVES.set('mirror', ({ el, ctx, wake, cleanup }) => {
    const output = document.createElement('output');
    output.setAttribute('$text', 'label');
    el.after(output);

    wake(output, ctx);
    cleanup(() => output.remove());

    return () => {};
});

Custom directives cannot create new row scopes in the current API. They can decorate an element, show or hide DOM, create DOM under the context they receive, and clean up after themselves. Scope-introducing behavior, such as rendering one child context per item, is reserved for built-ins like *list until a public scope helper exists.

A directive receives a DirectiveContext object:

FieldPurpose
path, isRel, parent, paramsThe parsed binding string, using the same lookup syntax as $ bindings.
hostThe DOM id host from a //id/path source, otherwise empty.
protocolA reserved parser field for future custom resolvers.
elThe element carrying the directive attribute.
ctxThe current binding context.
wake(node, ctx)Wires directive-created DOM, including any bindings or child wrappers inside it.
cleanup(off)Registers teardown work with the current context owner.

The public directive exports are DW_DIRECTIVES, DirectiveContext, DirectiveHandler, DirectiveUpdater, and nearestItem(ctx). Low-level row construction and owner subscription arrays are internal; custom directives should use wake() and cleanup() instead.
