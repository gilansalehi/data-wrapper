Reactivity model

data-wrapper does not track individual property reads. Instead, each mounted wrapper remembers which bindings are connected to the DOM. When a flush runs, the wrapper re-reads those bindings and updates the DOM only where the value changed.

That makes the model simple: mutate your state, then make sure a flush happens. Event bindings do this for you. Code that runs outside an event binding can use action() or call flush() manually. 

<script type="module" data-module="@view/counter">
    export let count = 0;
    export const doubled = () => count * 2;

    export function increment() {
        count += 1;
    }
</script>

<button @click="increment">+1</button>
<output $text="count"></output>
<output $text="doubled"></output>

When the button is clicked, the exported increment function runs inside the event action boundary. After it returns, data-wrapper re-reads count and doubled. A function used as a value is called during that read, so derived values can stay as ordinary JavaScript functions.

action()

Use action() for exported writers that may be called from outside a local @event binding. This is common when one module imports a writer from another module, or when application code calls a writer directly. 

import { action } from 'data-wrapper';

export let todos = [];

export const addTodo = action(task => {
    todos = [...todos, { id: Date.now(), task, done: false }];
});

action(fn) returns a wrapped function. The wrapped function calls your function, then schedules a global flush. Multiple action calls in the same task are coalesced, so a chain of writers still produces one flush at the end of the microtask. All mounted wrappers that read affected exports get a chance to re-read their active bindings. 

import { action } from 'data-wrapper';

export let todos = [];

export const { addTodo, removeTodo } = action({
    addTodo(task) {
        todos = [...todos, { id: Date.now(), task, done: false }];
    },
    removeTodo(id) {
        todos = todos.filter(todo => todo.id !== id);
    },
});

The object form wraps each function and returns the same shape. Calling action() on an already wrapped function returns the same function, so passing a writer through multiple modules does not multiply flushes.

Async actions

If an action returns a Promise, data-wrapper flushes after the synchronous part and again when the Promise settles. That covers the common loading pattern: show a loading state, await the data, then show the result. 

import { action } from 'data-wrapper';

export let loading = false;
export let items = [];

export const loadItems = action(async () => {
    loading = true;

    items   = await fetch('/items').then(r => r.json());
    loading = false;
});

flush() as an escape hatch

Some code runs outside the framework's event boundary and outside a writer you can wrap. Timers, third-party listeners, and browser APIs may mutate state later. In those cases, call flush() after the mutation. 

import { flush } from 'data-wrapper';

export let clock = new Date();

setInterval(() => {
    clock = new Date();
    flush();
}, 1000);

flush() is global. It asks every active wrapper runtime to re-read its connected bindings. Each runtime compares new values with the previous values using Object.is, then publishes only the bindings that changed.
