Composition with *src

A page often needs more than one view. The wrapper's src attribute mounts the first view, and *src lets that view reserve a place for another source of content. That source can be another view file, or it can be child content the host page already wrote inside the wrapper.

Think of *src as an outlet. The directive lives on a <template>, resolves a normal binding name, and then fills that position with the resolved value. A non-empty string loads a child view. Captured child nodes are moved into place. If the value is missing or empty, the template body is rendered as fallback content. 

<script type="module" data-module="@layout/docs">
    export default ({ props, slots }) => ({
        heading: props.heading,
        view: props.view,
        toc: slots.toc,
    });
</script>

<main class="docs-page">
    <aside>
        <h1 $text="heading"></h1>
        <template *src="toc">
            <p>No table of contents yet.</p>
        </template>
    </aside>

    <article>
        <template *src="view">
            <p>No page selected yet.</p>
        </template>
    </article>
</main>

The factory is still the source of truth for what the template can bind. In this example, the layout exposes heading, view, and toc. The template does not reach into props or slots directly; the factory chooses which values become part of the layout's binding surface.

Compose by reference

When the resolved value is a string, *src treats that string as a view URL. The directive creates a child <data-wrapper> at the outlet and loads that view under the current binding context.

Treat *src values like code. A string value causes a view fetch and can execute that view's module script, so bind trusted application state, never user input. 

<data-wrapper
    src="/views/layout.html?heading=Framework&view=/views/pages/framework.html">
</data-wrapper>

The host passes view through the wrapper URL. The layout factory returns view: props.view, and the <template *src="view"> outlet loads the page view in that position. This gives you layout-style composition without a build step or a router layer.

Compose by value

A host can also write content inside the wrapper. data-wrapper captures those child nodes before the loaded view replaces the wrapper contents. A plain slot attribute is used only as a grouping key; this is light DOM projection, not Shadow DOM slotting. 

<data-wrapper src="/views/layout.html?heading=Framework">
    <ol slot="toc">
        <li><a href="#intro">Intro</a></li>
        <li><a href="#tokens">Three tokens</a></li>
    </ol>
</data-wrapper>

The layout receives those captured nodes as slots.toc. When the factory returns toc: slots.toc, the *src="toc" outlet moves the authored list into the layout. Any bindings inside projected content wake in the layout's binding context, so projected HTML can still use the values the layout exposes.

Fallback content

The template body is the fallback. It can contain more than one top-level element, and it only appears when the resolved value is missing, empty, or otherwise not loadable. 

<template *src="view">
    <h2>Choose a page</h2>
    <p>The selected view will appear here.</p>
</template>

A string is always interpreted as a view URL. If you want to render text, use $text. This keeps *src focused on composition: reserve a position, resolve a source, and fill the outlet.
