Where data-wrapper sits

Before comparing anything, be clear about what this is: data-wrapper is an independent, early-stage library, not a full-stack platform. It is one custom element and about 1,400 lines of TypeScript — 18 kB minified, ~7 kB gzipped, zero dependencies, no build step. This page compares that focused design to mature frameworks with large teams, companies, and communities behind them. The adoption risks are real, but so is the leverage when the fit is right.

The model, in one paragraph

A <data-wrapper src="/views/x.html"> fetches a plain HTML view. The view's <script type="module"> is a real ES module: its named exports are the component's state and actions, and an optional default factory receives props, slots, and cleanup for per-instance scope. Three attribute tokens wire the markup to those exports — $prop binds a value, @event dispatches to an exported function, and *directive on a <template> controls structure. There are exactly three directives: *list (keyed rows), *if (conditional block), and *src (composition — load a view by URL, or project passed-in nodes).

How it knows data changed

Every framework's personality comes from this one question. There are roughly four answers in the wild:

  • Compile it away — a compiler analyzes assignments and emits surgical DOM updates (Svelte).
  • Track reads — signals or proxies record who read what, then notify exactly those readers (Solid, Vue, Alpine).
  • Re-render and diff — re-run the component, diff a virtual DOM, patch the difference (React, Mithril).
  • Re-read on a boundary — after a known event, re-read watched values and publish the ones that changed (Angular's classic zone-based change detection is the famous ancestor).

data-wrapper takes the fourth road, with a small surface: every binding subscribes to a named export. When an exported action runs (or you call flush()), each live component re-reads its subscribed exports, compares with Object.is, and publishes only the values that changed. No dependency graph, no proxies, no virtual DOM, no compiler. Mutate plain data, and the flush boundary finds it. The cost is proportional to active bindings — cheap for pages and dashboards. Animation-rate state, streaming updates, and very large grids need finer-grained machinery (see the gaps below).

Bindings are URLs

The binding expression language is not a new syntax — it is URL grammar. A bare name walks the scope chain; ./field reads the current list row; ../name steps out to the parent row; /name pins to the component root; //other-id/name reads another wrapper on the page. Query params chain formatters: $text="price?currency=EUR", *list="todos?sort=-created&key=id". One grammar, parsed by the platform's own URL class.

Guardrails on by default

$innerhtml throws — raw HTML requires the explicit $unsafeHTML opt-in. Bound URL attributes strip javascript: schemes. Views load same-origin only. Event handlers go through @event; $onclick-style bindings are blocked. An unresolvable binding renders as its literal text and warns in the console instead of failing silently.
