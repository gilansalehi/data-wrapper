What you give up for good

These gaps are consequences of the core bet — no build step, no compiler, no dependency tracking. They are permanent by definition: if any one of them is a dealbreaker for your project, no future release of data-wrapper fixes it, and you should pick from the right-hand side of the table above.

Templates are invisible to TypeScript — forever

Bindings are strings in HTML attributes, and having no compiler is the product. Nothing will ever check at build time that $text="todoCont" should have been todoCount — at runtime it renders the literal text and warns in the console, which is honest but late. Angular, Svelte, and Vue type-check templates during compilation; JSX frameworks get it free from TypeScript itself. If your team leans on the compiler to refactor safely, this is the gap that bites hardest, and it is out of our hands by design.

Coarse-grained invalidation is the model

A flush re-reads every subscribed export of every live component and diffs with Object.is. There is no dependency graph to narrow the work, and there never will be — that's the trade that keeps the whole reactivity model explainable in one paragraph. At page scale it's invisible. Streaming data, large keyed grids, or animation-rate state belong with Solid, Svelte, or Vue, which update exactly what changed.

No server rendering, no hydration

Every wrapper fetches its view after the page loads, so wrapper content is invisible to first paint and to crawlers that don't execute JavaScript. A static prerender helper could soften first paint someday, but a real SSR-and-hydration protocol is a framework-scale commitment that contradicts the ship-files philosophy. If your dynamic content is SEO-critical, Next, Nuxt, SvelteKit, and Angular solved this years ago.

Ecosystem of one

No Stack Overflow tag, no hiring pool, no LTS policy, and a bus factor of one. This isn't a design decision — but no roadmap manufactures a community either. React's real product is not the virtual DOM; it's the ten thousand packages and the person you can hire tomorrow who already knows it. Nothing at hobby scale competes with that, and this page won't pretend to.

What you're only giving up today

These are gaps of the current release, not of the design — each one has a plausible path that doesn't break the no-build, three-token contract. Read this as the de-facto roadmap. No dates and no promises: pre-1.0 means it ships when it's true.

Router

*src already swaps views when its bound value changes, so a router is not a framework feature — it's a component: a module that reads location, exposes the matched view URL as an export, and handles history on navigation. Adopting today means wiring that yourself; it does not mean SPAs are out of reach.

Devtools

The runtime already keeps a registry of every live component, its subscribed exports, and their last published values. An inspector panel can therefore be — what else — a data-wrapper view that lists them. Until it exists, debugging is the console plus the DOM inspector, which stays workable because the DOM you inspect is the DOM you wrote.

A component catalog

Views are plain HTML files, so shareable drop-in components (nav drawer, tabs, dialog, data table) need publishing and documentation, not library changes. The showcase widgets on this site are the seed of exactly that.

Contract edges

Several current limitations are closable rather than structural: bindings inside SVG subtrees, multi-root *if/*list bodies (the machinery already exists for *src fallback content), a richer lifecycle surface behind the factory context, and first-class testing helpers. Each can land incrementally without new grammar.
