What you give up for good

These gaps are consequences of the core bet — no build step, no compiler, no dependency tracking. They are structural tradeoffs: if one is a hard requirement for your project, a framework built around that requirement is the better fit.

Templates are runtime-checked, not type-checked

Bindings are strings in HTML attributes, and having no compiler is the product. The browser will not check at build time that $text="todoCont" should have been todoCount. data-wrapper surfaces the mistake at runtime by rendering the literal text and warning in the console. Angular, Svelte, and Vue can type-check templates during compilation; JSX frameworks lean on TypeScript in the JavaScript layer. If your team relies on compile-time template checks to refactor safely, this is the main tradeoff.

Coarse-grained invalidation is the model

A flush re-reads every subscribed export of every live component and diffs with Object.is. There is no dependency graph to narrow the work — that's the trade that keeps the whole reactivity model explainable in one paragraph. At page scale it's invisible. Streaming data, large keyed grids, or animation-rate state belong with Solid, Svelte, or Vue, which update exactly what changed.

No server rendering, no hydration

Every wrapper fetches its view after the page loads, so wrapper content is invisible to first paint and to crawlers that don't execute JavaScript. A static prerender helper could soften first paint someday, but a real SSR-and-hydration protocol is a framework-scale commitment that contradicts the ship-files philosophy. If your dynamic content is SEO-critical, Next, Nuxt, SvelteKit, and Angular are designed for that job.

Early ecosystem, single maintainer

data-wrapper does not yet have a Stack Overflow tag, a hiring pool, an LTS policy, or a community of maintainers. That is not a flaw in the design, but it is an adoption fact. React's real product is not only the virtual DOM; it is the packages, tutorials, conventions, and people already trained to use it. data-wrapper has to earn that trust over time.

What you're only giving up today

These are gaps of the current release, not of the design — each one has a plausible path that doesn't break the no-build, three-token contract. Read this as the de-facto roadmap. No dates and no promises: pre-1.0 means it ships when it's true.

Router

*src already swaps views when its bound value changes, so a router is not a framework feature — it's a component: a module that reads location, exposes the matched view URL as an export, and handles history on navigation. Adopting today means wiring that yourself; it does not mean SPAs are out of reach.

Devtools

The runtime already keeps a registry of every live component, its subscribed exports, and their last published values. An inspector panel can therefore be — what else — a data-wrapper view that lists them. Until it exists, debugging is the console plus the DOM inspector, which stays workable because the DOM you inspect is the DOM you wrote.

A component catalog

Views are plain HTML files, so shareable drop-in components (nav drawer, tabs, dialog, data table) need publishing and documentation, not library changes. The showcase widgets on this site are the seed of exactly that.

Contract edges

Several current limitations are closable rather than structural: bindings inside SVG subtrees, multi-root *if/*list bodies (the machinery already exists for *src fallback content), a richer lifecycle surface behind the factory context, and first-class testing helpers. Each can land incrementally without new grammar.
