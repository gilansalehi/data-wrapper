Security

data-wrapper's security posture is measured against published standards rather than an ad-hoc list: the OWASP DOM-Based XSS Prevention Cheat Sheet, the CWE-79 weakness taxonomy, CSP Level 3, and the Trusted Types API. This section states what the framework guarantees, what it leaves to the application, and where it does not yet reach.

Scope & threat model

The current contract assumes developer-authored views. Normal text bindings are safe for arbitrary strings because $text writes textContent, and URL-shaped bindings get a runtime scheme guard. Rendering user-generated HTML, accepting user-authored view files, and loading third-party views are out of scope today: an app that renders untrusted HTML owns sanitization until the Trusted-Types tier lands (see below).

A loaded view is trusted code. When load() fetches a view, it parses that HTML and imports the view's data-module script with full page privileges. Two rules follow from that: never flow user input into a wrapper src or a *src binding, and never serve user-uploaded files from the application origin. If uploads live at /uploads/..., an attacker-controlled upload can become a same-origin view candidate. Serve uploads from a separate origin.

Default protections

  • Safe-by-default text. $text writes textContent; bound values are never parsed as HTML.
  • Gated raw HTML. $innerHTML and $outerHTML throw. Raw HTML requires the explicit $unsafeHTML opt-in, so the dangerous sink can never be reached by accident.
  • URL-scheme allowlist. A bound value with a scheme is written only if that scheme is affirmatively allowed for the attribute; unknown schemes are dropped with a console warning, and the check resists control-character obfuscation such as java script:. Schemeless values (relative paths, //host, query, fragment) always pass. The matrix: http: / https: on every URL-bearing attribute (href, src, action, formaction, data, ping, poster, background); mailto: and tel: additionally on href; data: and blob: additionally on src and poster for media embedding. The src allowance is attribute-based, so a data: iframe src passes — it renders with an opaque origin and cannot reach the page. An exotic legitimate scheme can still be authored statically; the guard covers bound values only.
  • Event handlers only via @event. Binding an $on* property throws and points to the @event interface, so the event-handler sink cannot be reached through a $ binding.
  • No string evaluation. No eval, Function, or string-argument timers. Events bind functions, not code strings; JSON uses JSON, not eval.
  • Same-origin view loading. A <data-wrapper src> only loads views from the page's origin. A string rendered through *src creates a child wrapper and goes through the same origin gate. The value is still trusted code input; bind application state, not user input.

Application responsibilities

Within the stated scope the framework closes the common DOM-XSS gotchas; these remain the application's to own:

  • Treat the dangerous sinks as trusted-input-only: src and *src load and execute view code; $unsafeHTML and $srcdoc write raw HTML.
  • Sanitize backend data before binding it into an HTML sink. URL bindings enforce a scheme allowlist, but they do not decide whether a normal https: destination is appropriate for the app.
  • Do not bind untrusted input to $style. CSS injection is not script execution in modern browsers, but it can still deface UI or enable browser-specific data exfiltration tricks.
  • Set defense-in-depth the framework cannot: HttpOnly cookies, Subresource Integrity on the CDN framework script, and the CSP directives noted above.
