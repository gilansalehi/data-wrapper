Security

data-wrapper's security posture is measured against published standards rather than an ad-hoc list: the OWASP DOM-Based XSS Prevention Cheat Sheet, the CWE-79 weakness taxonomy, CSP Level 3, and the Trusted Types API. This section states what the framework guarantees, what it leaves to the application, and where it does not yet reach.

Scope & threat model

The current contract assumes developer-authored views. Normal text bindings are safe for arbitrary strings because $text writes textContent, and URL-shaped bindings get a runtime scheme guard. Rendering user-generated HTML, accepting user-authored view files, and loading third-party views are out of scope today: an app that renders untrusted HTML owns sanitization until the Trusted-Types tier lands (see below).

Default protections

  • Safe-by-default text. $text writes textContent; bound values are never parsed as HTML.
  • Gated raw HTML. $innerHTML and $outerHTML throw. Raw HTML requires the explicit $unsafeHTML opt-in, so the dangerous sink can never be reached by accident.
  • URL-scheme neutralization. javascript: and vbscript: values are dropped from every URL-bearing attribute (href, src, action, formaction, data, ping, poster, background), and the check resists control-character obfuscation such as java script:.
  • Event handlers only via @event. Binding an $on* property throws and points to the @event interface, so the event-handler sink cannot be reached through a $ binding.
  • No string evaluation. No eval, Function, or string-argument timers. Events bind functions, not code strings; JSON uses JSON, not eval.
  • Same-origin view loading. A <data-wrapper src> only loads views from the page's origin, limiting injected-src escalation.

OWASP DOM-XSS Cheat Sheet conformance

Cheat Sheet guidancedata-wrapper
RULE #6 / #7, GUIDELINE #3 — populate the DOM with safe properties (textContent) Met by default: bindings write DOM properties and textContent, never parsed markup.
GUIDELINE #4 — avoid innerHTML / outerHTML / document.write Enforced: bare HTML-sink bindings throw; raw HTML requires the named $unsafeHTML opt-in.
RULE #5 — untrusted data in a URL attribute context Met across the URL-attr set: dangerous schemes are neutralized, including control-character-obfuscated forms.
RULE #3, GUIDELINE #5 / #10 — no event-handler or implicit eval sinks Met: no eval / Function; binding an event handler via $on* is rejected — events use the @event interface only.

Application responsibilities

Within the stated scope the framework closes the common DOM-XSS gotchas; these remain the application's to own:

  • Do not pass untrusted content to $unsafeHTML or to an <iframe $srcdoc> — both are raw-HTML sinks by definition.
  • Sanitize backend data before binding it into an HTML sink. URL bindings block executable schemes, but they do not decide whether a normal https: destination is appropriate for the app.
  • Set defense-in-depth the framework cannot: HttpOnly cookies, Subresource Integrity on the CDN framework script, and the CSP directives noted above.
