The pitch

Lightweight

One custom element. Three wiring tokens. No compiler, no VDOM, no component runtime. Add the script and start binding HTML.

Instantly familiar

State is visible as data-*. Events are browser events. Paths look like URLs. DOM properties stay DOM properties.

Minimal cognitive load

You do not learn a template language. You learn three prefixes: $ for data, * for structure, @ for events.

Most frontend tools ask you to move your UI into JavaScript. data-wrapper moves just enough reactivity into HTML that the page can explain itself. The browser remains the platform. The framework is the smallest useful agreement between state, markup, and events.

Start with one wrapper

Wrap a piece of HTML. Put state on the wrapper. Bind DOM properties with $. Delegate events with @. 

<script type="module" src="/dist/data-wrapper.js"></script>

<data-wrapper id="counter" data-count="0">
  <button @click="count/down">-</button>
  <output $text="/count"></output>
  <button @click="count/up">+</button>
</data-wrapper>

<script type="module">
  counter.register({
    'count/down': () => counter.put('count', n => Number(n) - 1),
    'count/up':   () => counter.put('count', n => Number(n) + 1),
  });
</script>

That is the whole loop: HTML declares what it needs, JavaScript handles what happened, and the wrapper keeps state reflected in the DOM.

The mental model

QuestionAnswer
Where does state live? On the wrapper's data-* attributes.
How does state reach the DOM? $prop="/key" sets el[prop].
How does state change structure? *if mounts/removes one node; *list renders rows from a template.
How do events reach app code? @click="topic" emits a topic handled with register().
What happens on update? The wrapper broadcasts the changed key to already-compiled subscribers.

The important part is what you do not think about: component lifecycles, render functions, dependency arrays, stores, build tools, hydration boundaries, or a second model of the page. The DOM is the page. The wrapper is the stateful boundary around the part you want to make reactive.

Why data-* works

Attributes are inspectable, serializable, and already meaningful to CSS. Using them as state is not a gimmick; it lets the platform carry more of the work.

  • Debugging is obvious. Open DevTools and read the state on the element.
  • CSS becomes reactive. data-wrapper[data-filter="done"] is a state selector.
  • Server HTML starts alive. Initial state can ship in markup.
  • Persistence is simple. Read from localStorage, a server payload, or inline HTML, then put().

If your UI can be described as "some state changes some DOM", this is often enough.

Where it fits

data-wrapper is not trying to replace every frontend tool. It is for the broad middle where you want a reactive page, but you do not want to adopt an application framework.

ToolGreat when...data-wrapper differs by...
Alpine You want inline expressions and local component behavior. Keeping behavior in registered event topics and state in visible data-*.
htmx The server owns interactions and returns HTML fragments. Handling client-local state without requiring a request for every UI change.
Stimulus You want controller classes attached to existing markup. Using declarative state bindings instead of imperative DOM targets.
React/Vue/Svelte You need a component system, ecosystem, and build-time tooling. Letting existing HTML be the source instead of compiled component output.

A useful shorthand: htmx makes HTML talk to the server. Alpine makes HTML execute local expressions. data-wrapper makes HTML subscribe to wrapper state.

Good fits

Server-rendered pages

Add reactive islands without replacing the page architecture. The wrapper can start from server-provided data-*.

Internal tools

Forms, tables, filters, dashboards, CRUD widgets. Most of the value comes from predictable bindings and low ceremony.

Embeddable widgets

Drop into an existing page without asking what framework the host app uses. The boundary is a custom element.

Documentation demos

Examples stay honest because the markup is live. The code you show is the code the browser runs.

How it works in one pass

  1. The wrapper connects and wakes its subtree.
  2. wake() compiles $, *, and @ attributes once.
  3. Bindings become subscriber functions stored on the wrapper or row.
  4. put(), patch(), push(), and pull() update state.
  5. The wrapper broadcasts the changed key to the subscribers that already exist.

Runtime updates do not re-parse templates or walk the page. The setup cost happens at wake time; state changes call closures.

The constraints are the feature

The framework stays small because it refuses to become a second browser. It does not own routing, animation, fetch policy, validation, or your app architecture. It owns one thing: connecting wrapper state to DOM output.

  • If browser events already solve it, use browser events.
  • If CSS selectors already solve it, use CSS selectors.
  • If HTML already describes it, keep it in HTML.
  • If a tiny subscriber can update it, do not build a render engine.

That is the bet: the less framework you have to learn, the more of the platform you get to keep.
